# Preparing a GnosisVPN NetVM

SEQS can build `A-gnosisvpn`, a network-provider AppVM prepared for a later
manual GnosisVPN installation. The component installs only Debian prerequisites
and Qubes integration. It does not download, authenticate, or install
GnosisVPN.

## Build the prepared qube

After fetching, reviewing, and staging this revision, run in dom0:

```bash
~/s.sh --build-only --qubes gnosisvpn
```

This creates `Z-gnosisvpn` and `A-gnosisvpn`, enables
`provides_network=True`, and enables the Qubes firewall service.

The template already contains both prerequisites:

- `wireguard-tools`, including `wg` and `wg-quick`;
- `openresolv`, including the `resolvconf` command GnosisVPN expects.

Do not install `openresolv` again. GnosisVPN itself is intentionally absent.

`A-gnosisvpn` is already the prepared AppVM and is the shortest path for the
test below. To use a separately named AppVM instead, create it from
`Z-gnosisvpn`, then configure it in dom0 (replace `sys-gnosisvpn` with the
chosen exact name):

```bash
qvm-prefs sys-gnosisvpn provides_network true
qvm-service --enable sys-gnosisvpn qubes-firewall
qvm-run -u root sys-gnosisvpn /usr/sbin/seqs-gnosisvpn-prepare-app
```

The last command installs the persistent AppVM firewall hook from inherited
template assets. Do not skip it: manually creating an AppVM does not run the
SEQS app role that prepared `A-gnosisvpn`.

## Manual GnosisVPN test

For an initial same-session test, start `A-gnosisvpn` and manually install and
run GnosisVPN there according to the independently reviewed vendor procedure.
Do not attach sensitive client qubes yet.

An ordinary AppVM's root filesystem is reset from its template at shutdown.
Consequently, a package manually installed under `/usr` or `/etc` inside
`A-gnosisvpn` is suitable only for the current test session and will normally
disappear when the AppVM shuts down. SEQS does not work around that property.
Once the test is satisfactory, a later reviewed component should install the
GnosisVPN package in `Z-gnosisvpn`; alternatively, use a deliberately created
StandaloneVM if persistent manual system-package installation is required.

Do not replace or relink `/etc/resolv.conf`. It should remain Qubes-managed:

```text
nameserver 10.139.1.1
nameserver 10.139.1.2
```

When GnosisVPN brings up `wg0_gnosisvpn`, `wg-quick` registers its resolver
addresses under `/run/resolvconf/interfaces/wg0_gnosisvpn`. A systemd path
unit notices that runtime change. `/usr/sbin/seqs-gnosisvpn-dns` validates the
provider and QubesDB IPv4 addresses and atomically maintains a separately owned
`ip seqs_gnosisvpn_dns` nftables table:

- the output chain translates DNS generated by `A-gnosisvpn`;
- the prerouting chain translates DNS arriving from downstream client qubes.

The output rules apply only when routing selected `wg0_gnosisvpn`. On
disconnect, the helper removes its table. The Qubes firewall hook also blocks
forwarded IPv4 and IPv6 fallback through `eth0`, so attached clients fail
closed while the VPN is unavailable. The GnosisVPN qube itself retains
upstream access so the VPN software can discover and establish its tunnel.

## Verify before attaching clients

With GnosisVPN disconnected:

```bash
getent hosts example.com
sudo nft list table ip seqs_gnosisvpn_dns
```

Ordinary Qubes DNS should work; the nft command should report that the table
does not exist.

Connect GnosisVPN, then verify:

```bash
sudo wg show wg0_gnosisvpn
sudo resolvconf -l
cat /etc/resolv.conf
getent hosts example.com
sudo nft list table ip seqs_gnosisvpn_dns
```

The WireGuard interface should have a recent handshake, `resolvconf -l` should
show one or two IPv4 resolvers for `wg0_gnosisvpn`, `/etc/resolv.conf` should
still contain Qubes' synthetic addresses, name resolution should succeed, and
the nftables counters should increase.

In another terminal, confirm that translated DNS crosses the tunnel:

```bash
sudo tcpdump -ni wg0_gnosisvpn 'port 53'
sudo tcpdump -ni eth0 'port 53'
```

Trigger a lookup. Provider DNS should appear on `wg0_gnosisvpn`, not `eth0`.

Only then attach a non-sensitive disposable test client to `A-gnosisvpn`.
Inside that client:

```bash
cat /etc/resolv.conf
getent hosts example.com
curl https://ifconfig.co
```

The client should retain Qubes' synthetic resolver addresses, resolve names,
and report the VPN egress address. Disconnect GnosisVPN and confirm that the
client loses both DNS and IP connectivity instead of falling back to the
upstream NetVM. Reconnect and repeat the checks before attaching other qubes.

## Caveats

- This repository does not verify or install GnosisVPN yet. A manual download
  extends trust to the vendor, its distribution channel, and the exact
  procedure used.
- The DNS helper expects the interface and openresolv entry to be named
  `wg0_gnosisvpn`. A future GnosisVPN release that renames either requires a
  reviewed component update.
- Only IPv4 DNS servers are accepted. An IPv6-only resolver announcement makes
  the helper remove its table and fail rather than interpolate unsupported
  data.
- Forwarded IPv6 is fail-closed through `eth0`. This setup does not claim
  working IPv6 through GnosisVPN.
- A manually created AppVM requires `provides_network`, the
  `qubes-firewall` service, and one successful root invocation of
  `seqs-gnosisvpn-prepare-app`; the generated `A-gnosisvpn` already has all
  three.
- The DNS path unit reacts to openresolv runtime-file changes. Check
  `systemctl status seqs-gnosisvpn-dns.path` and
  `journalctl -u seqs-gnosisvpn-dns.service` if the VPN connects but the table
  is absent.
- A compromised GnosisVPN qube can change its own routes and nftables rules.
  Put a separate firewall qube downstream if enforcement must remain outside
  the VPN qube's trust boundary.
